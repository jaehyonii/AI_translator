# 2026 글로벌 공생 생성형 AI 영어번역 경시대회용 LLM 번역 아키텍처 설계

## 1. 설계 목표

이 경시대회는 생성형 AI를 활용해 초벌 번역을 수행한 뒤, 사람이 세밀하게 교정하여 최종 번역문을 제출하는 방식이다. 따라서 목표는 단순히 원문을 기계적으로 번역하는 것이 아니라, 대회 평가 기준에 맞춰 번역 품질을 최대화하는 **LLM 기반 번역 보조 아키텍처**를 설계하는 것이다.

평가 기준은 다음과 같다.

| 평가 항목 | 배점 | 아키텍처상 목표 |
|---|---:|---|
| 유창성 | 50점 | 목표 언어 독자가 자연스럽게 읽을 수 있는 번역문 생성 |
| 문체성 | 30점 | 대화체/서술체, 격식, 어휘 수준, 문장 구조를 원문에 맞게 유지 |
| 정확성 | 20점 | 원문 의미, 정보, 시제, 인물관계, 뉘앙스를 빠짐없이 반영 |

따라서 이 시스템의 핵심은 **정확한 초벌 번역기**가 아니라 **평가 기준 최적화형 번역 시스템**이다.

---

## 2. 전체 아키텍처 개요

```text
[시험 문항 입력]
      ↓
[전처리 모듈]
- 문단 분리
- 대사/서술 분리
- 고유명사/숫자/핵심 표현 표시
      ↓
[맥락 분석 Agent]
- 인물관계
- 줄거리
- 화자 의도
- 문체/톤
- 문화적 표현
      ↓
[번역 Router]
- 영한 대화체
- 영한 서술체
- 한영 대화체
- 한영 서술체
      ↓
[초벌 번역 Agent]
- 의미 보존 중심 번역
      ↓
[자연스러움 개선 Agent]
- 목표 언어답게 문장 재구성
      ↓
[문체 보정 Agent]
- 대화체/서술체에 맞게 톤 조정
      ↓
[정확성 검수 Agent]
- 누락, 오역, 과잉 의역, 시제, 인물관계 확인
      ↓
[평가 기준 Scoring Agent]
- 유창성/문체성/정확성 기준으로 자체 채점
      ↓
[최종 편집본]
      ↓
[사람이 마지막 검토 후 제출]
```

하나의 고성능 LLM을 여러 역할별 프롬프트로 나누어 사용할 수 있으며, 반드시 여러 모델을 사용할 필요는 없다.

## 3. 핵심 데이터 구조: Context Pack

번역 품질을 안정적으로 유지하려면, 먼저 원문에 대한 맥락 정보를 구조화해야 한다. 이 정보를 **Context Pack**이라고 부른다.

```json
{
  "direction": "Korean to English / English to Korean",
  "text_type": "dialogue / narrative",
  "summary": "전체 내용 요약",
  "characters": [
    {
      "name": "인물명",
      "role": "화자/청자/서술 대상",
      "relationship": "친구/상사/교수/학생/가족/연인 등",
      "tone": "친근함/격식 있음/냉소적/불안함 등"
    }
  ],
  "plot_flow": [
    "사건 1",
    "사건 2",
    "사건 3"
  ],
  "style": {
    "register": "formal / casual / literary / academic",
    "voice": "calm / emotional / persuasive / humorous",
    "sentence_length": "short / medium / long"
  },
  "key_terms": [
    {
      "source": "원문 표현",
      "translation": "권장 번역",
      "note": "주의점"
    }
  ],
  "risk_points": [
    "오역 가능성이 높은 표현",
    "문화적 의역이 필요한 표현",
    "직역하면 어색한 문장"
  ]
}
```

Context Pack은 이후 번역, 문체 보정, 정확성 검수의 기준이 된다.

## 4. 모듈별 설계

### 4.1 입력/전처리 모듈

#### 역할

- 원문을 그대로 보존한다.
- 문단 번호를 부여한다.
- 대화체라면 화자별로 구분한다.
- 고유명사, 숫자, 날짜, 장소, 직함을 표시한다.
- 긴 문장을 의미 단위로 나눈다.

#### 예시 출력

```text
[문단 1]
원문...

[문단 2]
원문...

[주의 요소]
- 인물: John, Mary
- 관계: 직장 동료
- 장소: 회의실
- 핵심 사건: 프로젝트 마감 연기
```

---

### 4.2 맥락 분석 Agent

#### 역할

번역을 바로 수행하지 않고, 원문을 먼저 분석한다.

#### 분석 항목

| 요소 | 설명 |
|---|---|
| 인물관계 | 누가 누구에게 말하는지 파악 |
| 줄거리 | 사건의 흐름 또는 논리 전개 파악 |
| 화자 의도 | 부탁, 거절, 설득, 농담, 비꼼, 사과 등 구분 |
| 감정선 | 불안, 분노, 당황, 기쁨, 냉소 등 파악 |
| 문체 | 격식체, 일상체, 문학적 서술체, 설명문체 등 구분 |
| 문화 표현 | 직역하면 어색한 표현 탐지 |
| 핵심 어휘 | 반복되는 용어, 고유명사, 중요한 단어 정리 |
| 위험 문장 | 중의적 표현, 긴 문장, 관용구 표시 |

---

### 4.3 번역 Router

대회 문항은 네 가지 유형으로 나뉜다.

1. 영한번역 - 대화체
2. 영한번역 - 서술체
3. 한영번역 - 대화체
4. 한영번역 - 서술체

각 유형마다 다른 번역 프롬프트와 전략을 사용한다.

```text
if direction == "EN→KO" and type == "dialogue":
    use E2K_DIALOGUE_PROMPT

if direction == "EN→KO" and type == "narrative":
    use E2K_NARRATIVE_PROMPT

if direction == "KO→EN" and type == "dialogue":
    use K2E_DIALOGUE_PROMPT

if direction == "KO→EN" and type == "narrative":
    use K2E_NARRATIVE_PROMPT
```

## 5. 유형별 번역 전략

### 5.1 영한번역 - 대화체

#### 핵심 목표

영어 대화를 한국어 대화처럼 자연스럽게 옮긴다.

#### 체크 요소

- 존댓말/반말
- 말투의 거리감
- 감정 표현
- 생략과 호흡
- 영어식 표현 제거
- 대명사 반복 줄이기

#### 예시

```text
I didn't say you were wrong.
```

직역:

```text
나는 네가 틀렸다고 말하지 않았다.
```

자연스러운 대화체:

```text
네가 틀렸다고 한 건 아니야.
```

---

### 5.2 영한번역 - 서술체

#### 핵심 목표

영어 문장의 구조를 그대로 옮기기보다, 한국어 글로서 자연스럽게 재구성한다.

#### 체크 요소

- 영어식 어순 제거
- 긴 문장 쪼개기
- 자연스러운 접속어 사용
- 문단 흐름 유지
- 서술자의 시선 유지
- 문학적/설명적 분위기 보존

#### 예시

```text
As the city slowly woke beneath a pale morning sky, he realized that nothing would ever be the same again.
```

가능한 번역:

```text
희미한 아침 하늘 아래 도시가 서서히 깨어나고 있을 때, 그는 이제 모든 것이 예전과 같을 수 없다는 사실을 깨달았다.
```

---

### 5.3 한영번역 - 대화체

#### 핵심 목표

한국어에서 생략된 정보를 영어에서 자연스럽게 복원한다.

#### 체크 요소

- 생략된 주어 추가
- 시제 명확화
- 영어식 대화 표현 사용
- 한국어식 직역 피하기
- 높임말을 영어의 공손한 표현으로 변환
- 감정의 세기 조절

#### 예시

```text
잠깐 얘기 좀 할 수 있을까요?
```

상대가 교수나 상사라면:

```text
Could I speak with you for a moment?
```

친구라면:

```text
Can we talk for a second?
```

---

### 5.4 한영번역 - 서술체

#### 핵심 목표

한국어 문장을 영어 논리 구조에 맞게 재구성한다.

#### 체크 요소

- 주어와 동사 명확화
- 문장 간 논리 연결
- 관사, 단수/복수 확인
- 시제 일관성 유지
- 한국어식 긴 문장 분해
- 영어다운 표현 선택

#### 예시

```text
그는 오랫동안 미뤄 두었던 결정을 내려야 할 때가 왔다는 것을 느꼈다.
```

자연스러운 영어:

```text
He felt that the time had come to make the decision he had long put off.
```

## 6. Agent 구성 상세

### 6.1 Context Analyzer Agent

#### 목적

원문을 번역하기 전에 분석한다.

#### 출력

- 요약
- 인물관계
- 줄거리
- 문체
- 감정
- 번역 위험 요소
- 용어집

#### 프롬프트 예시

```text
너는 전문 번역가의 분석 보조자다.
아래 원문을 번역하지 말고, 번역에 필요한 맥락만 분석하라.

분석 항목:
1. 글의 종류: 대화체/서술체
2. 전체 요약
3. 등장인물과 관계
4. 사건의 흐름
5. 화자의 의도와 감정
6. 문체와 톤
7. 번역 시 주의할 표현
8. 고유명사, 숫자, 날짜, 핵심 용어
9. 직역하면 어색해질 문장

원문:
[여기에 원문 입력]
```

---

### 6.2 Draft Translation Agent

#### 목적

정확성을 우선으로 초벌 번역을 만든다.

#### 특징

- 원문 정보 누락 금지
- 과도한 의역 금지
- 문단 구조 유지
- 고유명사, 숫자, 관계 보존

#### 프롬프트 예시

```text
너는 영어-한국어 전문 번역가다.
아래 Context Pack을 반영하여 원문을 번역하라.

우선순위:
1. 원문의 의미를 빠짐없이 보존한다.
2. 문단 구조를 유지한다.
3. 고유명사, 숫자, 시제, 인물관계를 정확히 반영한다.
4. 아직 과도하게 문학적으로 다듬지 말고, 정확한 초벌 번역을 만든다.

Context Pack:
[분석 결과]

원문:
[원문]
```

한영번역일 경우 첫 문장을 다음과 같이 바꾼다.

```text
너는 한국어-영어 전문 번역가다.
```

---

### 6.3 Fluency Refiner Agent

#### 목적

평가 기준 중 가장 큰 비중인 유창성 50점을 끌어올린다.

#### 기능

- 목표 언어다운 문장으로 재구성
- 어색한 직역 제거
- 문장 길이 조절
- 자연스러운 연결어 추가
- 반복 표현 줄이기

#### 프롬프트 예시

```text
아래 번역문을 목표 언어 독자가 자연스럽게 읽을 수 있도록 다듬어라.

조건:
1. 원문 의미를 바꾸지 말 것
2. 누락이나 추가 정보를 만들지 말 것
3. 문체는 Context Pack과 일치시킬 것
4. 직역투를 제거할 것
5. 대화체라면 실제 사람이 말하는 듯하게, 서술체라면 글답게 다듬을 것

Context Pack:
[분석 결과]

초벌 번역:
[초벌 번역]
```

---

### 6.4 Style Controller Agent

#### 목적

문체성 30점을 담당한다.

#### 대화체 보정 기준

- 자연스러운 말투
- 관계에 맞는 공손함
- 감정 표현
- 말의 호흡
- 너무 설명적이지 않은 표현

#### 서술체 보정 기준

- 문단의 흐름
- 서술자의 거리감
- 어휘 수준
- 문장 리듬
- 문학적/논리적 분위기

#### 프롬프트 예시

```text
아래 번역문을 문체 기준에 맞게 보정하라.

문체 기준:
- 유형: [대화체/서술체]
- 번역 방향: [영한/한영]
- 톤: [격식 있음/일상적/문학적/설명적/감정적 등]
- 인물관계: [관계 정보]
- 독자에게 자연스럽게 읽히는 수준으로 조정할 것

주의:
- 의미를 바꾸지 말 것
- 과도하게 화려하게 만들지 말 것
- 문체만 개선할 것

번역문:
[번역문]
```

---

### 6.5 Accuracy Verifier Agent

#### 목적

원문과 번역문을 비교하여 정확성 오류를 찾는다.

#### 검수 항목

- 원문 문장별 의미 대응
- 누락된 정보
- 추가된 정보
- 고유명사 오류
- 숫자/날짜 오류
- 시제 오류
- 인물관계 오류
- 대명사 오류
- 관용구 오역
- 부정문 오역
- 조건문 오역

#### 프롬프트 예시

```text
너는 번역 검수자다.
원문과 번역문을 비교하여 오류를 찾아라.

검수 항목:
1. 누락된 정보
2. 추가된 정보
3. 오역
4. 시제 오류
5. 인물관계 오류
6. 고유명사/숫자/날짜 오류
7. 문체 불일치
8. 직역투 또는 부자연스러운 표현

출력 형식:
- 문제 위치
- 문제 유형
- 설명
- 수정 제안
- 심각도: 높음/중간/낮음

원문:
[원문]

번역문:
[번역문]
```

---

### 6.6 Rubric Scorer Agent

#### 목적

대회 평가 기준에 맞춰 번역문을 자체 평가한다.

#### 프롬프트 예시

```text
아래 번역문을 다음 기준으로 평가하라.

평가 기준:
1. 유창성 50점: 목표 언어로 자연스럽게 읽히는가
2. 문체성 30점: 문체, 구조, 어휘가 원문과 목적에 맞는가
3. 정확성 20점: 원문 내용이 빠짐없이 반영되었는가

각 항목별로:
- 점수
- 감점 이유
- 개선 제안
- 반드시 고쳐야 할 문장

원문:
[원문]

번역문:
[번역문]
```

이 Agent의 목적은 실제 점수 예측이 아니라 마지막 수정 포인트를 찾는 것이다.

## 7. 전체 실행 흐름

```text
1. 문항 원문 입력
2. Context Analyzer 실행
3. 유형 확인
   - 영한 대화체
   - 영한 서술체
   - 한영 대화체
   - 한영 서술체
4. 초벌 번역 생성
5. 유창성 개선
6. 문체 보정
7. 정확성 검수
8. 오류 반영 후 재작성
9. 평가 기준으로 자체 채점
10. 사람이 최종 검토 후 제출
```

### 의사코드

```python
for problem in problems:
    source_text = load_problem(problem)

    context_pack = analyze_context(source_text)

    route = classify_route(
        direction=context_pack["direction"],
        text_type=context_pack["text_type"]
    )

    draft = translate_draft(
        source_text=source_text,
        context=context_pack,
        route=route
    )

    fluent_version = refine_fluency(
        translation=draft,
        context=context_pack
    )

    styled_version = control_style(
        translation=fluent_version,
        context=context_pack
    )

    qa_report = verify_accuracy(
        source_text=source_text,
        translation=styled_version,
        context=context_pack
    )

    final_translation = revise_with_qa(
        translation=styled_version,
        qa_report=qa_report
    )

    score_report = score_by_rubric(
        source_text=source_text,
        translation=final_translation
    )

    human_review(final_translation, score_report)
```

| 항목 | 확인 질문 |

| 감정 | 원문의 감정과 뉘앙스가 살아 있는가 |
| 의미 | 원문 정보가 빠지지 않았는가 |
| 문체 | 대화체/서술체 구분이 명확한가 |

## 10. 대화체 전용 체크리스트

대화체는 특히 인물관계가 중요하다.

LLM에게 반드시 분석시켜야 할 요소는 다음과 같다.

```text
1. 두 사람은 어떤 관계인가?
2. 나이/지위 차이가 있는가?
3. 친한 사이인가, 공식적인 사이인가?
4. 화자는 부탁하는가, 따지는가, 농담하는가?
5. 말투는 부드러운가, 날카로운가?
6. 감정이 드러나는 표현은 어디인가?
7. 번역문에서 존댓말/반말을 어떻게 처리할 것인가?
```

대화체 번역에서 피해야 할 것은 다음과 같다.

```text
- 지나치게 문어적인 표현
- 영어식 직역
- 어색한 존댓말
- 감정 없는 평면적 번역
- 모든 문장을 완전한 문장으로 번역하는 것
```

예를 들어 영어 대화에서 다음 문장은 상황별로 다르게 번역될 수 있다.

```text
Are you serious?
```

가능한 번역:

```text
정말이야?
진심이야?
장난하는 거야?
정말 그렇게 생각해?
```

## 11. 서술체 전용 체크리스트

서술체는 흐름과 리듬이 중요하다.

LLM에게 반드시 분석시켜야 할 요소는 다음과 같다.

```text
1. 글의 목적은 설명, 묘사, 주장, 회상 중 무엇인가?
2. 문체는 문학적인가, 학술적인가, 일상적인가?
3. 서술자의 태도는 객관적인가, 감정적인가?
4. 핵심 사건의 순서는 무엇인가?
5. 문단 간 논리 연결은 어떻게 되는가?
6. 반복되는 핵심 어휘는 무엇인가?
7. 은유나 상징 표현이 있는가?
```

서술체 번역에서 피해야 할 것은 다음과 같다.

```text
- 대화체처럼 너무 가볍게 번역
- 문장을 과하게 쪼개서 리듬을 깨는 것
- 원문의 분위기를 없애는 것
- 영어 문장 구조를 그대로 한국어에 옮기는 것
- 한국어 문장을 그대로 영어 어순으로 옮기는 것
```

## 12. 실전 시간 운영 설계

대회 시간은 18:30부터 20:00까지 총 90분이다. 문항은 4개이므로 한 문항당 평균 20분 내외로 처리해야 한다.

### 추천 시간 배분

```text
전체 준비 및 문항 확인: 5분

문항 1: 18분
문항 2: 18분
문항 3: 18분
문항 4: 18분

최종 전체 검토: 8분
업로드 및 제출 확인: 3분
예비 시간: 3분
```

### 문항별 세부 시간

```text
1. 원문 입력/정리: 1분
2. 맥락 분석: 2분
3. 초벌 번역: 3분
4. 유창성/문체 보정: 5분
5. 정확성 검수: 3분
6. 직접 최종 수정: 4분
```

핵심은 AI에게 너무 오래 재번역을 시키지 않는 것이다. 한 문항당 `초벌 → 개선 → 검수 → 최종` 정도로 제한하는 것이 좋다.

## 13. 최종 제출 전 사람 검토 기준

AI가 만든 번역을 그대로 제출하면 위험하다. 마지막에는 사람이 아래 항목을 직접 검토해야 한다.

### 13.1 소리 내어 읽었을 때 자연스러운가

유창성 점수와 직결된다.

```text
이 문장을 실제 사람이 말할까?
이 문장을 실제 글에서 볼 수 있을까?
```

### 13.2 원문과 의미가 같은가

특히 아래 요소를 확인해야 한다.

```text
부정문
조건문
비교 표현
숫자
날짜
인물 이름
감정 표현
```

### 13.3 말투가 일관되는가

대화체에서 존댓말과 반말이 어색하게 섞이면 감점 위험이 크다.

### 13.4 너무 AI스럽지 않은가

AI 번역은 종종 다음과 같은 특징을 보인다.

```text
- 지나치게 깔끔하지만 밋밋함
- 모든 문장이 비슷한 길이
- 감정 표현이 약함
- 한국어인데 영어식 논리 구조가 남음
- 영어인데 한국어식 완곡 표현이 남음
```

## 15. 최종 추천 구조

대회용으로는 지나치게 복잡한 시스템보다 다음 구조가 가장 현실적이다.

```text
LLM 1개 + 프롬프트 4개 + 사람이 최종 편집
```

구체적인 운영 방식은 다음과 같다.

```text
1. 원문 붙여넣기
2. 맥락 분석 프롬프트 실행
3. 초벌 번역 프롬프트 실행
4. 검수 프롬프트 실행
5. 최종 제출본 프롬프트 실행
6. 사람이 마지막으로 자연스럽게 수정
```

복잡한 멀티 Agent 시스템도 가능하지만, 시험장에서는 시간이 제한되어 있으므로 **속도, 안정성, 반복 가능성**이 더 중요하다.

## 16. 결론

이 대회에서 좋은 결과를 얻기 위한 LLM 아키텍처는 단순 번역기가 아니라 다음과 같은 구조여야 한다.

```text
맥락 분석형 번역 시스템
+ 문체 제어 시스템
+ 정확성 검수 시스템
+ 평가 기준 기반 최종 편집 시스템
```

핵심은 다음 세 가지다.

```text
1. 번역 전에 인물관계, 줄거리, 화자 의도, 문체를 먼저 분석한다.
2. 초벌 번역 후 유창성과 문체를 따로 개선한다.
3. 마지막에는 원문과 비교해 누락, 오역, 어색한 직역투를 검수한다.
```

최종 실전 구조는 다음과 같다.

```text
Context Analyzer
→ Translator
→ Fluency/Style Refiner
→ Accuracy Checker
→ Final Human Editor
```

이 구조는 대회 평가 기준인 **유창성 50점, 문체성 30점, 정확성 20점**에 직접 대응하는 번역 시스템이다.
